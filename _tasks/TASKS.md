# Task Management

This folder holds human-authored project tasks for the 3T Bible Translation Platform.

The separate `_dev-tasks/` folder is generated by `_dev-system`; those files are advisory maintenance signals and should not replace this main task list.

## Task Types

### Main Project Tasks

- Use a three-digit prefix: `001_task_name.md`, `002_task_name.md`, etc.
- Choose the next number by checking the highest existing number in `_tasks/`.
- Make each task self-contained: objective, scope, constraints, deliverables, and acceptance criteria.
- Keep tasks focused on user-visible platform work, source-data workflow, editorial workflow, EPUB output, or project hygiene.

### Dev-System Tasks

- Location: `_dev-tasks/`
- Prefix: `D001`, `D002`, etc.
- Source: generated by `_dev-system/analyzer.py`.
- Purpose: identify files that may need refactoring, tighter tests, or structural cleanup.
- These are advisory. Do not treat a large file as automatically wrong if it is a stable orchestration boundary.

### Troubleshooting Tasks

- Use prefix `T` plus the next three-digit task number when a debugging session needs a written trail, for example `T009_fix_chat_streaming_regression.md`.
- Include the observed failure, reproduction steps, suspected files, and verification command.

## Current Task Index

The ordered index is `_tasks/000_INDEX.md`.

When adding, completing, or retiring a task, update `_tasks/000_INDEX.md` in the same change.

## Workflow

1. Pick one task from `_tasks/000_INDEX.md` or one advisory task from `_dev-tasks/`.
2. Read the task and inspect the relevant code before editing.
3. Keep edits scoped. Avoid unrelated refactors, data churn, or formatting sweeps.
4. Preserve compatibility shims in `src/ttt_webapp/` unless all imports, tests, and launch scripts are migrated.
5. For browser workbench changes, prefer active modules:
   - `src/ttt_workbench/webapp.py`
   - `src/ttt_workbench/controller.py`
   - `src/ttt_workbench/templates/`
   - `src/ttt_workbench/static/`
6. Do not reintroduce the deprecated auth proxy. Use the direct llama.cpp endpoint configuration.
7. Do not commit `.env`, `.ttt_workbench/`, local databases, archives, generated EPUBs, or runtime output.

## Verification

Use the smallest useful check while developing, then run the full project check before commit when practical.

Quick syntax check:

```bash
.venv/bin/python -m compileall -q src tests _dev-system/analyzer.py
```

Focused test:

```bash
.venv/bin/python -m pytest tests/test_webapp_fake_mode.py -q
```

Full test suite:

```bash
./ttt.sh test
```

Dev-system scan:

```bash
./_scripts/run_dev_analyzer.sh
```

## Completion

For a completed task:

1. Update the task file with a short `Status` or completion note if useful.
2. Update `_tasks/000_INDEX.md`.
3. Keep the task file in `_tasks/` unless the project later adopts `completed/`, `pending/`, and `postponed/` subfolders.
4. Mention verification results in the commit or final work summary.

## Folder Policy

- `_tasks/`: main human-authored project tasks.
- `_dev-tasks/`: generated advisory maintenance tasks.
- `_dev-system/`: analyzer implementation, config, and reports.
- `_scripts/`: small project maintenance runners.
